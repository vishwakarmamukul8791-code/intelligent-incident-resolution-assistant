                                                         INTERVIEW QUESTION 

Why do we chunk? 
ans -  LLMs and embedding models work better on smaller semantic units.

Large documents are split into chunks so that embeddings can be generated for each chunk independently, improving retrieval accuracy during semantic search.

Why not generate one embedding for the entire PDF?
ans - A single embedding represents the whole document and loses fine-grained context.

By splitting the document into chunks and generating embeddings for each chunk, the retrieval system can find the most relevant section of the document instead of returning the entire document.
wrong ans - Because the document is large.

Why UploadFile instead of bytes?
ans- UploadFile streams the file and is memory efficient.
bytes loads the complete file into memory.


Why store metadata along with embeddings?

Embeddings are only vector representations. After similarity search, the system must retrieve the original chunk text and metadata associated with the matched vector. Therefore embeddings are stored together with references to the source content.

Why do we use: IndexFlatL2

A normal list requires checking every embedding one by one.

FAISS is optimized for fast similarity search on large vector datasets containing thousands or millions of embeddings.

Why maintain metadata with embeddings?
ans - Embeddings enable similarity search, while metadata helps retrieve the original chunk text, source document, page number, and other contextual information after retrieval.


Why not store chunk text inside FAISS?- 
FAISS is optimized for vector similarity search.
Document text and metadata are stored separately and retrieved using the vector index returned by FAISS.

Why store source filename?
After retrieval, the support engineer should know which document produced the answer.

How are documents converted into vectors in your RAG project?
ans -> We use Sentence Transformers (all-MiniLM-L6-v2), a BERT-based embedding model.

The model converts each document chunk into a dense semantic vector of 384 dimensions. These vectors are then stored in a FAISS index for similarity search.
Sentence Transformers capture semantic meaning, not just vocabulary matching.

So words like:

login
sign in
authenticate

can produce similar vectors even though the exact words are different.

TF-IDF relies mainly on word occurrence and frequency, so if the vocabulary doesn't match, similarity may be low.
Embeddings capture contextual meaning of text, allowing semantic search, whereas TF-IDF relies on exact keyword overlap.
faiss.write_index(index, path)
→ saves vector index to disk

faiss.read_index(path)
→ loads vector index from disk

This allows embeddings to persist across application restarts.

Why use hash instead of filename?-> Filenames can change while content remains the same.
Hashing is based on the actual file content, allowing duplicate-content detection and efficient re-indexing.


FAISS index is loaded from disk if it exists.
New embeddings are appended using index.add().
The updated index is persisted back to disk using faiss.write_index().

Why save metadata before FAISS? -> FAISS stores only vectors.
Metadata stores the original chunk text and document information.
After FAISS returns a matching vector index, metadata is used to retrieve the actual chunk content.

What is Retrieval in RAG? 
ans -> Retrieval is the process of finding the most relevant chunks from the knowledge base using vector similarity search before sending context to the LLM.

What does FAISS return? 
FAISS returns vector indexes and similarity scores.
It does not return the actual text chunks.

Why is metadata stored in the same order as vectors?
ans -> The FAISS vector position and metadata position must match so that retrieved vector indexes can be mapped back to the original text chunks.

Why are embeddings converted to float32?
ans ->FAISS expects vectors in float32 format.
Using other data types may cause errors or inefficient memory usage.

What is K in Top-K Search? -> K is the number of nearest neighbors returned by the vector database.

Why do we use indices[0] -> FAISS returns results as a 2D array because it supports searching multiple queries at once.

Since we search one query, the first row contains our results.

What happens if duplicate embeddings are stored in FAISS? -> FAISS will return multiple highly similar results because the same content exists multiple times in the vector index.

Why not directly send all documents to the LLM? -> Documents may contain thousands of chunks and exceed the LLM context window. Retrieval fetches only the most relevant chunks, reducing cost and improving answer quality.

Q: What is Context in RAG? -> Context is the collection of retrieved chunks passed to the LLM along with the user query to generate an informed response.
